### Minor: 2

#### MINOR-1: `_APPROX_PRICE_USD` fallback in dashboard route is stale

**File:** `backend/main.py:724`
**Section:** Dashboard endpoint, fallback price for `USDBTC`

When CoinGecko is unreachable, line 724 uses `1.0 / 62000.0` as the USD→BTC fallback. This corresponds to BTC at $62,000. Current BTC price (from CoinGecko defaults in monitor) is $105,000. The stale fallback means the BTC-denominated portfolio value shown on the dashboard will be off by ~70% during the first ~120s after a cold start.

**Impact:** Low — this is a cold-start transient; prices refresh within 120s. But if CoinGecko is permanently unreachable, the fallback persists indefinitely.

**Fix:**
```python
_usd_btc = _prices.get("USDBTC", 1.0 / 105000.0)  # match monitor's _APPROX_PRICE_USD["btc"]
```

#### MINOR-2: `list_wallets` endpoint uses `w.get()` which may fail on SELECT * results

**File:** `backend/main.py:856-859`

The `list_wallets` endpoint uses `w.get("balance_usd")` on asyncpg `Record` objects. While asyncpg Records do support `.get()` (they behave like dicts), this is less common than dict-style access. Not a bug — works correctly — but worth noting for consistency with the rest of the codebase which uses direct key access `w["balance_usd"]`.

**Impact:** None — functional. Style observation only.

The startup handler at `main.py:166-173` shows:
```python
except Exception as e:

```

When DB is unreachable:
1. `db_pool = None` (caught exception)

3. No wallet polling → prices never refresh → stale defaults persist
4. Health endpoint reports `db.ok: false`, `monitor.alive: false`, `price_cache.fresh: false`

This is all correct graceful degradation. The fix requires resolving the DB connection issue in Zeabur (check DATABASE_URL env var, verify PostgreSQL service is running in the same project).
