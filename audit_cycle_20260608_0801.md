## ✅ Work Completed

### 1. New Migration: `migrations/009_performance_indexes.sql`

**What it does:**
Adds two composite indexes on `copy_trade_signals`:

| Index | Columns | Covers |
|-------|---------|--------|
| `idx_copy_trade_signals_wallet_created` | `(wallet_id, created_at)` | 30d/90d signal count filters, median amount calculation |
| `idx_copy_trade_signals_wallet_status_created` | `(wallet_id, status, created_at)` | Execution rate calculation (`status='executed' + created_at`) |

**Why this matters:**
The `whale_scorer.score_whale_wallet()` query runs once per whale wallet per monitor cycle (every 60s). It filters `copy_trade_signals` by `(wallet_id, created_at)` with multiple WHERE clauses:
- `WHERE cts.created_at >= NOW() - INTERVAL '30 days'` (signal count 30d, median amount)
- `WHERE cts.created_at >= NOW() - INTERVAL '90 days'` (signal count 90d)
- `WHERE cts.status = 'executed' AND cts.created_at >= NOW() - INTERVAL '30 days'` (execution rate)

Without these indexes, PostgreSQL must do a **full table scan** on `copy_trade_signals` for each wallet. With 10 whale wallets and 10K signals, that's 10 full scans per cycle. The indexes reduce this to narrow index range scans.

**Root cause of the gap:**
The indexes were created in `backend/migrations/013_copy_trade_signals_performance_indexes.sql` but were never consolidated into the root `migrations/` directory. The root migrations are what Zeabur runs on deploy, so the production database has been missing these indexes.
