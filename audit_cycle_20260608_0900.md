## ✅ Work Completed

### 1. New Endpoint: `GET /api/wallets/{wallet_id}/score`

**File:** `backend/main.py` (+71 lines)

**What it does:**
- Authenticated endpoint (requires JWT via `get_current_user`)
- Verifies wallet ownership (user can only score their own wallets)
- Pre-computes global median once (same optimization as monitor Phase 6)
- Runs `whale_scorer.score_whale_wallet()` on-demand
- Returns full scoring breakdown including:
  - All 5 sub-scores: activity, reliability, weight, recency, diversity
  - Cold-start indicators: `score_is_coldstart`, `score_signals_used`
  - Signal stats: `median_amount_30d`, `execution_rate_30d`
  - DB comparison: `db_stored_score` vs live score, `db_score_calculated_at`

**Why this matters:**
- The monitor updates whale scores every 60s in the background, but operators cannot see *why* a wallet has a particular score
- This endpoint makes the scoring transparent and debuggable
- The `db_stored_score` vs live score comparison helps detect stale scores (when the monitor is not running)

**Design decisions:**
- Uses separate DB connections for wallet lookup, global median, and scoring (no connection held across HTTP calls)
- Follows existing patterns: `acquire_db()`, `get_current_user`, `HTTPException`
- Reuses `score_whale_wallet()` directly (no code duplication)
- Global median pre-computation avoids N+1 subqueries
