## ✅ Work Completed

### 1. New Migration: `migrations/010_migration_log_and_tx_composite_index.sql`

**What it does:**

| Object | Type | Purpose |
|--------|------|---------|
| `_migration_log` | Table | Tracks applied migrations (id, migration_id, file_name, applied_at, checksum, execution_time_ms) |
| `idx_migration_log_id` | Index | Fast lookup by migration_id |
| `idx_migration_log_applied_at` | Index | Ordered scan for health diagnostic |
| `idx_transactions_wallet_timestamp` | Composite index | Covers `JOIN wallets ON wallet_id + ORDER BY timestamp DESC` pattern |

**Why the composite index matters:**
The activity feed and dashboard recent-transactions queries both use:
```sql
SELECT t.* FROM transactions t
JOIN wallets w ON w.id = t.wallet_id
WHERE w.user_id = $1
ORDER BY t.timestamp DESC
LIMIT 20
```
With separate indexes on `wallet_id` and `timestamp`, PostgreSQL must combine them via bitmap scan. The composite index `(wallet_id, timestamp DESC)` enables a single narrow index range scan, reducing I/O for this hot query path.

**Root cause of the _migration_log gap:**
The table was defined in `check_migration_status.py` (line 172-186) as `MIGRATION_LOG_DDL` and created lazily via `_ensure_migration_table()`, but was never consolidated into the root `migrations/` directory that Zeabur runs on deploy. This meant:
- The table only exists if `check_migration_status.py` was run first
- The health diagnostic endpoint's migration summary always showed an error
- Manual migration runners had no tracking

**Commit:** `58720f2`
