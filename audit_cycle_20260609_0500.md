## ✅ Work Completed

### 1. Per-Phase Timing Instrumentation (`services/monitor.py`)

**Commit:** `24ef25d`

**What was added:**

Six phase timing measurements in `_poll_all_wallets_inner()`:

| Phase | What it measures | Typical range |
|-------|-----------------|---------------|
| `phase1_read_wallets` | DB query to fetch all wallet rows | 1-5ms |
| `phase2_prune_state` | In-memory state cleanup | <1ms |
| `phase3_fetch_wallets` | Concurrent on-chain balance/tx fetches | 2-30s |
| `phase4_collect_changes` | Result aggregation | <1ms |
| `phase5_batch_update` | DB write transaction | 5-50ms |
| `phase6_signals_alerts` | Signal generation, alerts, WS pushes | 10-500ms |

**How the data is exposed:**
- Each cycle's `phase_durations_s` dict is stored in the `_cycle_stats` ring buffer (exposed via `get_cycle_stats()` → `/api/health`)
- INFO log on every cycle: `Monitor cycle complete: X.Xs | phases: phase1=0.002s, phase2=0.001s, ...`
- WARNING log if any phase exceeds 5s threshold (configurable via `_PHASE_WARN_THRESHOLD_S`)

**Design decisions:**
- Phase timings use `time.monotonic()` (immune to clock skew)
- No external dependencies — pure monotonic clock arithmetic
- Zero overhead when logging is below INFO level (dict comprehension is cheap)
- `_phase_durations` dict is reset each cycle (no unbounded growth)
- Threshold constant is local to the function (not a module-level config — avoids global mutable state)
