## ✅ Work Completed

### 1. Lock-Splitting in `_ensure_prices_fetched()` (`services/monitor.py`)

**Commit:** `fcfb6c7`

**Problem:**
The `_ensure_prices_fetched()` function held `_price_cache_lock` through the entire HTTP fetch to CoinGecko (two concurrent `asyncio.gather` calls). When CoinGecko is rate-limited (HTTP 429) or slow, the lock is held for the full timeout duration. Since every wallet check in Phase 3 calls `_ensure_prices_fetched()`, all concurrent wallet processing blocks on this lock — turning a single API issue into a full monitor stall.

**Fix: Three-phase lock splitting pattern:**

| Phase | What happens | Lock held? |
|-------|-------------|------------|
| A: Check staleness | Read cache timestamp, check if < 60s old | ✅ Yes (microseconds) |
| B: Fetch from CoinGecko | Two concurrent HTTP requests | ❌ No |
| C: Update cache | Write new prices + timestamp | ✅ Yes (microseconds) |

**Additional improvement — thundering herd prevention:**
A `_fetch_in_progress` flag in the price cache dict prevents multiple concurrent coroutines from all issuing redundant CoinGecko requests when they simultaneously detect a stale cache. Only one coroutine performs the fetch; others skip and use whatever the cache currently has.

**Error handling:**
- On fetch failure: the `_fetch_in_progress` flag is cleared under lock, so the next cycle can retry immediately
- On success: flag is cleared after updating the cache
- Both paths ensure the flag never gets stuck at `True`

**Design decisions:**
- No new module-level variables — the flag lives in the existing `_price_cache` dict
- All existing `.get()` callers are unaffected (they don't reference `_fetch_in_progress`)
- The `import time as _time_mod` is local to the function (avoids module-level import change)
- Lock hold time reduced from ~seconds (HTTP timeout) to ~microseconds (dict read/write)
