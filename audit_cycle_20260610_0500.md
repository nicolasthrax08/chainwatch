## ✅ Work Completed

### 1. Alert History `message` field — Backend + Frontend Fix

**Commit:** `7f71ce9`

**Problem:**
The `fired_alerts` table has a `message` column (added in migration `006_fired_alerts_message.sql`) that stores human-readable alert descriptions like `"Whale buy: ETH $15,000"` or `"Large transaction: USDC $50,000"`. However:

1. `get_alert_history()` in `main.py` did NOT SELECT the `message` column
2. The response dict did NOT include `message`
3. The frontend Alerts.jsx history table had no Message column
4. The `field_contract.py` registry was also missing the field

This meant users saw alert history with rule type, trigger value, and time — but not the human-readable description of what actually happened.

**Fix (3 files):**

| File | Change |
|------|--------|
| `backend/main.py` | Added `fa.message` to SQL SELECT, `'message': r["message"] or ""` to response dict |
| `backend/services/field_contract.py` | Added `'message'` to `fired_alert` field set in `alert_history` endpoint registry |
| `frontend/src/pages/Alerts.jsx` | Added `<th>Message</th>` header and `<td>{h.message || '—'}` cell with `maxWidth: 320` + ellipsis overflow |

**Design decisions:**
- `r["message"] or ""` ensures NULL values render as empty string (not "null")
- Frontend uses `maxWidth: 320`, `whiteSpace: 'nowrap'`, `overflow: 'hidden'`, `textOverflow: 'ellipsis'` to handle long messages gracefully
- Message column placed after Rule Type, before Trigger Value (most useful ordering)
