## ✅ Work Completed

### 1. DB Connection Retry with DNS Pre-Resolution

**File:** `backend/main.py` (startup handler, ~line 255)

**Root Cause:**
The startup log shows: `'service-6a199cb95a225abf61ee099b' does not appear to be an IPv4 or IPv6 address`

This is asyncpg's error when it cannot resolve the hostname from DATABASE_URL. The Zeabur internal DNS resolution is transient — it may fail on the first attempt during container startup but succeed on retry. The original code had no retry mechanism, so a single DNS failure permanently disabled the DB pool for the entire container lifetime.

**Fix Applied:**
```python
# Retry loop: 3 attempts with exponential backoff (1s, 2s, 4s)
for _attempt in range(3):
    if _attempt > 0:
        _backoff = 2 ** (_attempt - 1)
        await asyncio.sleep(_backoff)
    
    # Pre-resolve DNS on each attempt
    if _db_host:
        try:
            _socket.getaddrinfo(_db_host, None)
        except Exception as _dns_err:
            if _attempt < 2:
                continue  # retry after backoff
    
    try:
        db_pool = await asyncpg.create_pool(DATABASE_URL, ...)
        break  # success
    except Exception as e:
        if _attempt == 2:
            db_pool = None  # all attempts failed
```

**Key design decisions:**
- **3 attempts** with exponential backoff (1s, 2s) — total max wait ~3s, acceptable for startup
- **DNS pre-resolution** before each attempt — warms up the DNS cache and catches resolution failures early
- **Per-attempt logging** — each failure is logged with attempt number for debugging
- **Startup log events** — `db_pool_attempt_failed` for each failure, `db_pool_created` with attempt number on success
- **Fail-closed** — if all 3 attempts fail, `db_pool = None` (same as before, but now with better diagnostics)
- **`application_name: "chainwatch"`** — added to server_settings for easier DB session identification
