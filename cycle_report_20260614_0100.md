### Changes Made

#### 1. Fixed duplicate migration ID 019 → 020
**File**: `backend/migrations/020_signal_confidence_final.sql` (renamed from `019_signal_confidence_final.sql`)

Two migrations shared the number 019:
- `019_wallets_unique_constraint.sql` — adds unique constraint on wallets(user_id, address)
- `019_signal_confidence_final.sql` — adds confidence_final column

The `check_migration_status.py` tool assigns migration IDs by extracting `-- Migration N` from the SQL header. Both files had `-- Migration 019`, producing identical `migration_019` IDs. Since `_migration_log.migration_id` has a UNIQUE constraint, whichever file was applied first would cause the other to be silently treated as "already applied."

Alphabetically, `019_signal_confidence_final.sql` comes after `019_wallets_unique_constraint.sql`, so the confidence_final migration would be silently skipped — meaning the `confidence_final` column (from the previous cycle's work) would never be created in the database.

**Fix**: Renamed the file to `020_signal_confidence_final.sql` and updated its `-- Migration 020:` header.

#### 2. Added `check_duplicate_migration_ids` to audit_source.py
**File**: `backend/audit_source.py` (lines ~4367-4425, registred at line 3952)

A new audit check that:
- Extracts migration IDs from all `.sql` files using the same logic as `check_migration_status.py`'s `_extract_migration_id()`
- Detects any duplicate IDs across migration files
- Reports duplicates as a critical finding with file list and fix suggestion

This prevents the entire class of duplicate-migration-ID bugs from going undetected in future.
